Purpose and Intended Audience

The intent of this document is to provide a high-level survey of key architectural components of the current implementation of an Integrated Clinical Environment devised at the Interoperability Lab in Cambridge, MA.  This implementation is exhibited to visitors to the lab and the software components are available to the public at https://www.openice.info.  This document offers readers a targeted glimpse of key design features of that implementation that pertain specifically to the Infusion Safety application.  The intended audience includes anyone who wishes to join the conversation about this implementation (and others).  Knowledge of software engineering is assumed.

Data Distribution Service

A word about DDS is in order.  The data distribution service is not yet another message-oriented middleware.  Rather it serves as a delegate for the task of managing the distributed state of a system composed of heterogeneous components that are loosely coupled yet highly aligned around a shared data model conception of the facets of the broader system state.  A cursory look at the API for DDS reveals no facilities for publishing or subscribing to streams of messages.  The message-passing paradigm, and its attendant need for strictly defined message passing semantics which often reduces to a proliferation of substates for participant as they move, in time, through a prescribed sequence of messages.  Rather an application is provided with Readers and Writers that maintain, on behalf of the application, a coherent view of the current state of homogeneous collections of state information.  (which is why the associated system data model, represented in IDL, is inherently normalized)

 

As an author of software that has received, by virtue of a sensor or by virtue of its own automata, updated state information to share with the system DDS allows one to clearly define their mechanism for sharing that information with other participants.  My sole responsibility with that data is to “write” the updated sample of a particular facet of system state to a Writer.  The Writer delegate then takes on responsibility for communicating that new sample to other participants under a system of constraints described in a robust set of Quality of Service parameters.

 

Conversely responsibilities delegated to the Reader include maintaining a coherent view of the state communicated by Writers under the terms of its own Quality of Service settings (and a reconciliation to ensure their compatibility with those QoS of a writer).  Simply by creating a Reader delegate and specifying a topic and type (the state managed by a single reader or writer is homogeneous) an application can access the current state of that facet of the system merely by reading samples from its own Reader delegate whenever it wants to examine state.  It is almost incidental that the Reader also provides a variety of highly granular semaphores for communicating events in its own duty cycle with other threads outside of the DDS pool.  For instance condition variables can be configured to awaken an external (to DDS) thread when new unread samples are available or when a writer has failed to pair with the reader because of incompatible Quality of Service parameters.  It is for convenience (though overly emphasized by introductory examples) that DDS also supports a direct observer pattern whereby listener callbacks are directly invoked by DDS threads to provide notifications but clearly this is less desirable as it creates an opportunity to stymie the functioning of the DDS abstraction by burning up arbitrary cycles on its workhorse threads which should be staying focused on the core task of maintaining state.

 

I previously mentioned that Readers and Writers maintain collections of homogeneous state information.  The mechanism that makes this possible is the selection of key fields of the data sample that distinguish that sample.  For instance the key values of a numeric observed value might include a nomenclature code.  By varying the components of the key values of data samples new “instances” are created in the DDS.  It is difficult to understand a lot of quality of service parameters without this concept.  Readers and Writers are effectively managing system state in two dimensions … across the domain of instance keys and across the temporal domain; storing samples of each instance through time.  So the Reader maintaining the state of numeric observed data can be interrogated for samples of data associated with a particular nomenclature element by requesting the samples of the associated instance.

Quality of Service

https://sourceforge.net/p/mdpnp/code/ci/0.5.0/tree/interop-lab/demo-apps/src/main/resources/META-INF/ice_library.xml

The strategy for the definition of default QoS parameters is to, by default, offer the most stringent possible settings on the default writer side and request the most lax possible settings on the default reader side.  Various inheriting profiles change settings to increase/decrease criticality for various types of topics (by profile).

I’d like to briefly explain the meaning of some of the QoS parameters I have chosen.

Liveliness

This setting can ask DDS to ascertain the liveliness of remote participants automatically (with its own heartbeat mechanism) by guaranteeing a remote participant has not remained silent for a period exceeding the lease duration.  While interrogating the liveliness of specific remote participants is possible it is not advised.  Rather data instances inherit their own liveliness from the writers associated with that instance.  So if an instance exists for observed blood temperature which has one writer and that writer is deemed unalive the instance is changed to the unalive state.  This allows me to focus on the data rather than thinking in a point-to-point manner about passing messages to other specific participants.

Default Profile

Reader

<liveliness>

<kind>DDS_AUTOMATIC_LIVELINESS_QOS</kind>

<lease_duration>

<sec>2147483647</sec>

<nanosec>2147483647</nanosec>

         </lease_duration>

</liveliness>

In the default case setting the maximum possible duration effectively disables requests for liveliness QoS from readers using the default configuration.  This maximizes the likelihood of successfully matching the offer of as many writers as possible.

Writer

<liveliness>

<kind>DDS_AUTOMATIC_LIVELINESS_QOS</kind>

<lease_duration>

                <sec>1</sec>

<nanosec>0</nanosec>

</lease_duration>

</liveliness>

In the default case setting a reasonably tight constraint of 1 second maximizes the likelihood of matching with a variety of readers.

observed_data Profile

Reader

<liveliness>

<kind>DDS_AUTOMATIC_LIVELINESS_QOS</kind>

<lease_duration>

<sec>5</sec>

<nanosec>0</nanosec>

</lease_duration>

</liveliness>

For observed data Readers specify a more stringent requirement that writers assert their liveliness every 5 seconds.  This means that if a device or sensor loses its connectivity to the reader notification can be given to applications reading those observations.

Durability

Durability configures the responsibility of writers to maintain samples even after they have been communicated to other interested participants.  As you can imagine this facilitates scenarios like the late-joiner by allowing a newly matched reader to receive from a writer the most recent samples of instances; even samples that were published before the reader became alive.  The setting here specifies that samples are maintained transiently (in memory only) and locally (by the originating writer).  Durability can also be facilitated by specialized participants that can make samples available from a persistent store if that is desirable.  From the reader perspective that happens transparently (it merely means more samples available for inspection).

Default Profile

Writer

<durability>

         <kind>DDS_TRANSIENT_LOCAL_DURABILITY_QOS</kind>

</durability>

The default writer offers to maintain a local (transient) record of samples bounded only by resource limits (on memory consumption).

Reader

 <durability>

         <kind>DDS_VOLATILE_DURABILITY_QOS</kind>

</durability>

The default reader requests volatile durability; not requiring writers to maintain any sample history.  Again maximizing the likelihood of matching with a writer QoS

state Profile

Reader / Writer

<durability>

         <kind>DDS_TRANSIENT_LOCAL_DURABILITY_QOS</kind>

</durability>

The “state” profile is used for various information representing state.  For example DeviceIdentity uses this QoS profile because it is desirable that the writer maintain DeviceIdentity information (which does not change regularly) to share with late joining readers.

observed_data profile

Reader / Writer

Similarly the observed_data profile (representing data sensed by a medical device) is expected to maintain a transient sample history (especially for the benefit of late joiners) who may like a short history of recent samples for display on the screen, for example.

History

The settings of the history QoS work in tandem with flavors of durability and specify the extent of the samples of each instance that should be maintained.

Default Profile

Reader

<history>

         <kind>KEEP_LAST_HISTORY_QOS</kind>

         <depth>500</depth>

</history>

The goal here is to request something less than ALL samples to increase likelihood of compatibility with writers.  VOLATILE default durability somewhat obviates this setting until other changes are made by inheritors from the default profile.

Writer

<history>

         <kind>KEEP_ALL_HISTORY_QOS</kind>

</history>

 By default writers offer the most stringent history; keeping all available samples bounded only by resource constraints.

state profile

Reader / Writer

<history>

         <kind>KEEP_LAST_HISTORY_QOS</kind>

         <depth>1</depth>

</history>

The default for state information is to maintain / request only the most recent state information.

Ownership

Default Profile

Reader / Writer

<ownership>

         <kind>DDS_EXCLUSIVE_OWNERSHIP_QOS</kind>

</ownership>

As previously mentioned instances are owned by writers that have registered to provide samples of that instance.  By this particular QoS setting I have indicated that only one writer may be registered to provide samples of an instance at a time.  As you will see later my numeric observed data instances are also keyed on UDI and therefore exclusivity makes sense; since there is a natural one-to-one correspondence between instances and devices.  This is not the only possible approach.

 

Publish Mode

Default Profile

Reader / Writer

<publish_mode>

         <kind>ASYNCHRONOUS_PUBLISH_MODE_QOS</kind>

</publish_mode>

Asynchronous publication allows my Writer delegate to return from the “write” function before data has necessarily been communicated to all participants.  This allows the underlying RTPS layer the flexibility to manage its own fragmentation of data.  Otherwise DDS will avail itself only of the features of the chosen transport mechanism.  Empirically I have learned that when sending, for instance, a ~64k raster DDS in synchronous mode will push that entire buffer off to the IP stack which will require it to be communicated with IP-level fragmentation … which is ill advised.  By allowing asynchronous publication RTPS can create datagrams tuned to fit within one frame.

heartbeat profile

Reader / Writer

<publish_mode>

         <kind>SYNCHRONOUS_PUBLISH_MODE_QOS</kind>

</publish_mode>

Heartbeat information is time sensitive and a small structure so it makes more sense to write it synchronously.

Reliability

As one would expect reliability specifies whether transmitted data need be affirmatively acknowledged and whether, in the absence of acknowledgment, transmission needs to be retried.  max_blocking_time is obviated by the aforementioned asynchronous writes but in the case of synchronous writing would bound the allowed time a write call can block whilst awaiting acknowledgment.  Here the acknowledgment of data samples has been delegated entirely to DDS with DDS_PROTOCOL_ACKNOWLEDGMENT_MODE. Alternatively the QoS of service might specify that the application itself must affirmatively acknowledge each sample through a function call exposed on the Reader side or that samples be acknowledged automatically only when they have been accessed by the reading application.

Default Profile

Reader

<reliability>

         <kind>DDS_BEST_EFFORT_RELIABILITY_QOS</kind>

</reliability>

Readers using the default profile request best effort (unacknowledged) data transmission to maximize the likelihood of compatibility with various writers.

Writer

<reliability>

         <kind>RELIABLE_RELIABILITY_QOS</kind>

         <max_blocking_time>

              <sec>2</sec>

              <nanosec>0</nanosec>

         </max_blocking_time>

         <acknowledgment_kind>DDS_PROTOCOL_ACKNOWLEDGMENT_MODE

         </acknowledgment_kind>

</reliability>

Writers using the default profile offer reliable (acknowledged) data transmission to maximize the likelihood of compatibility with various readers.

Highlights from the IDL definitions used between our device adapters and the infusion safety application.

Image information encoded as specified by a MIME content_type.

 

DeviceIdentity

  struct DeviceIdentity {

    UniqueDeviceIdentifier unique_device_identifier; //@key

    LongString manufacturer;

    LongString model;

    LongString serial_number;

    Image icon;

    string<128> build;

    string<128> operating_system;

  }; //@top-level true //@Extensibility MUTABLE_EXTENSIBILITY

Identifying information for a specific device.  Published by the device adapter at device adapter startup.  For several devices (including Nonin bluetooth pulse oximeter) this sample is republished when serial number is received from the device.  With a reader maintaining the latest sample of each instance of DeviceIdentity I can conveniently access identifying information for any given UDI.  By virtue of QoS configuration publishing this information one time makes it available to all late joiners so no re-publication is required.  Checking the liveliness of the instance associated with a UDI also checks for liveliness of the participating device (or device adapter).

DeviceConnectivity

  enum ConnectionState {

    Connected,

    Connecting,

    Negotiating,

    Disconnecting,

    Disconnected

  };

   

  enum ConnectionType {

    Serial,

    Simulated,

    Network

  };

struct DeviceConnectivity {

    UniqueDeviceIdentifier unique_device_identifier; //@key

    ConnectionState state;

    ConnectionType type;

    LongString info;

    ValidTargets valid_targets;

  }; //@Extensibility MUTABLE_EXTENSIBILITY

For further connected devices (i.e. device adapters with a secondary connection on to an actual device) device connectivity information is published.  For such adapters UDI refers to the adapter and not to the connected device… and the presence of a live DeviceIdentity instance does not fully reveal an active device connection.  For devices publishing instances of DeviceConnectivity the ConnectionState must also be examined.  In the PCA app this manifests in the UI as a red circle-slash when a device (adapter) reports it is in any state except for “Connected”.  Serially connected devices with unsophisticated protocols (lacking in presence indication messages) will deem the connection disconnected when data is not received for a specified “maximum quiet period”.  For serial connected devices the sequence valid_targets will be populated with available valid serial port names.  Many device adapters will not initiate a connection on to their actual device without an objective (see the next section).

  struct Numeric {

    UniqueDeviceIdentifier unique_device_identifier; //@key

    MetricIdentifier metric_id; //@key

    VendorMetricIdentifier vendor_metric_id; //@key

    InstanceIdentifier instance_id; //@key

    UnitIdentifier unit_id; //@key

    float value;

    Time_t device_time;

  }; //@top-level true //@Extensibility MUTABLE_EXTENSIBILITY

Observed numeric values within the system.  A single-precision floating point number is used to encode values.  The name field contains a nomenclature code specifying the type of the observed data (for instance MDC_PULS_OXIM_PULS_RATE [18458]).  The universal_device_identifier contains the UDI for the device that observed the data.  When a device (or adapter) receives new sensed data it’s sole responsibility is to publish that data to a topic for Numerics.

InfusionStatus  

  struct InfusionStatus {

    UniqueDeviceIdentifier unique_device_identifier; //@key

    // is the pump actively infusing?

    boolean infusionActive;

    // it's entirely unclear if patient id, care area, BSA, etc. should be here

    // measurement units are standardized to avoid inconsistent states

    // derived values are omitted to avoid inconsistent states

    // what's in the bag

    wstring<128> drug_name;

    long drug_mass_mcg;

    long solution_volume_ml;

    // concentration can be derived

    // what's the program

    long volume_to_be_infused_ml;

    long infusion_duration_seconds;

    // rate can be derived

    // what is the progress through the program?

    float infusion_fraction_complete;

  }; //@top-level true //@Extensibility MUTABLE_EXTENSIBILITY

The current status of infusions within the system.  Each pump, identified by UDI, may have its own status information. This is a first-pass and does not yet include active bolus information (as in actual PCA).

InfusionObjective

struct InfusionObjective {

    UniqueDeviceIdentifier unique_device_identifier; //@key

    LongString requestor;

    boolean stopInfusion;

}; //@top-level true //@Extensibility MUTABLE_EXTENSIBILITY

Current objectives for infusions within the system.  Each infusion is identified by UDI.  The participant requesting the objective be met is identified by the requestor field.  stopInfusion is a simple flag indicating whether the safety interlock should be activated and disallow an active infusion.  The infusion device monitors its related instance for samples of InfusionObjective and when stopInfusion becomes true stops the active infusion.  The controlling device (requestor) can determine whether the infusion has stopped by monitoring InfusionStatus.  stopInfusion=false does not imply the pump should be running.  stopInfusion=true indicates the pump must be stopped.  Interestingly the requestor could easily become a component of the instance key; allowing the distributed system to track the objectives of several devices for the particular infusion.  A participant, such as the infusion device itself, could then assign priorities or make heuristic determinations based on all the available information about which request should prevail.  Alternatively another participant might mediate a control relationship and publish that control information to another topic (to make it available to all participants).  Another alternative is the setting of the ownership strength QoS to ensure the appropriate one writer of objectives gains exclusive access to the instance.  Plan is to explore several of these options with respect to the PCA scenario.

Application Data Model

Because data are normalized in the DDS Data Model relationships exist that one might like to reflect in the containment tree of an object model.  In the DDS standard a Data Local Reconstruction Layer (DLRL) was proposed but has proven difficult to generalize in implementation.  In a sense this is the fundamental responsibility of the application and perhaps not worth generalizing anyway.  Another possible evolution here is to writing applications embrace the reader/writer paradigm for state management directly .. obviating the need for an “application data model” that is distinct from that of the system.  One can imagine that this single coherent model of system state would be more easily managed and less prone to failure.

For the time being, however, the application relays normalized DDS samples to an object model in memory by combining individual coherent pieces of state (data samples of a particular topic and type).  This might be residue of development in Swing (which adheres closely to an MVC worldview in which data models are object-oriented).  Further work is required to determine the feasibility of binding a UI to a normalized data model.

DeviceListModel

http://sourceforge.net/p/mdpnp/code/ci/master/tree/interop-lab/demo-apps/src/main/java/org/mdpnp/apps/testapp/DeviceListModel.java

The DeviceListModel binds instances of DeviceIdentity and DeviceConnectivity into an object model in the form of a Swing ListModel.  This model therefore furnishes the data model for the JList component listing connected devices in the UI.  This has proven useful for testing transient disconnects and has also revealed a subtle issue wherein DeviceIdentity instances are not considered “alive” upon restoration of connectivity until a new sample is published.

Value

http://sourceforge.net/p/mdpnp/code/ci/master/tree/interop-lab/demo-apps/src/main/java/org/mdpnp/apps/testapp/vital/Value.java

A value is an object representation of the latest Numeric sample of a particular vital sign from a particular source UDI.  Because a Value is contained within the context of a parent Vital it can expose methods indicating whether the vital has violated any constraints placed on all values for the vital sign.  Currently Value also tracks historical samples; but this behaviour should be pushed into the Numeric reader with a higher “keep last” setting in the history QoS settings.  The lifecycle of a Vital matches that of its associated Numeric instance.  When the Numeric instance becomes alive the Value is created and when the Numeric instance becomes “unalive” the Value is destroyed.  This is a simplification because instances can remain in a Reader in disposed or “no writers” states indefinitely (subject to resource constraints).  Further the UI could leverage this capability to reflect “last known” values with a visual indication that no further updates are expected from a dead instance.

Vital

http://sourceforge.net/p/mdpnp/code/ci/master/tree/interop-lab/demo-apps/src/main/java/org/mdpnp/apps/testapp/vital/Vital.java

A vital is an object representing a particular constrained vital sign in the Infusion Safety application.  The vital tracks, among other things, the name of the vital sign, the constraints in place for the vital sign, the nomenclature names deemed synonymous with this vital, and it collects the various available Values for the vital sign.  Vital lifecycle is controlled by the user in configuration of the application.  Vital has no direct analog in the DDS model but perhaps it should.  This information, tracked by the Infusion Safety App, might be of interest to other participants.

VitalModel

http://sourceforge.net/p/mdpnp/code/ci/master/tree/interop-lab/demo-apps/src/main/java/org/mdpnp/apps/testapp/vital/VitalModel.java#l36

The VitalModel contains all the information tracked by one instance of the Infusion Safety Application and is bound to the Infusion Safety UI at runtime.  In addition to serving as a container for Vitals the VitalModel also furnishes readers of DeviceIdentity and DeviceConnectivity, mostly to the UI, for random access to information useful for display purposes.  The VitalModel also tracks the current state of the application at every change, evaluating whether an alarm condition exists, and whether the infusion safety interlock is active.  The VitalModel is also responsible for publishing an interlock objective to the infusion device as necessary.
